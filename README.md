## Model

### `complaint_model.dart`

**Purpose:** The data blueprint for a complaint object. Ensures consistent field naming between the app and the Supabase database.

**Class:** `Complaint`

**Fields:**

| Field | Type | Notes |
|---|---|---|
| `id` | `int?` | Auto-generated by DB |
| `userId` | `String` | FK → `profiles.id` |
| `categoryId` | `int` | FK → categories table |
| `title` | `String` | Complaint title |
| `description` | `String` | Full complaint text |
| `status` | `String` | Default: `'pending'` |
| `createdAt` | `DateTime?` | Parsed from ISO string |
| `studentName` | `String?` | From join/view |
| `categoryName` | `String?` | From join/view |

**Factory Constructor:**
```dart
factory Complaint.fromMap(Map<String, dynamic> map)
```
Safely parses all fields with null fallbacks. `createdAt` is parsed via `DateTime.parse()`.

## Services

### `auth_service.dart`

**Purpose:** Handles all Supabase Auth operations — registration, login, role-based routing, and logout.

**Class:** `AuthService`

#### Methods

**`signUp({email, password, fullName})`**
- Calls `_supabase.auth.signUp()`
- Sets `role: 'student'` in user metadata by default
- Throws `AuthException.message` on failure

**`signIn(email, password)`**
- Calls `_supabase.auth.signInWithPassword()`
- Throws error string on failure

**`getUserRole()`** → `Future<String>`
- Queries `profiles` table for the current user's `role`
- Returns `'student'` as default fallback if no record found
- Used by `LoginScreen` to determine routing destination

**`currentUser`** → `User?`
- Getter for `_supabase.auth.currentUser`

**`signOut()`**
- Calls `_supabase.auth.signOut()`

---

### `admin_complaint_service.dart`

**Purpose:** Data Access Object (DAO) for all admin-facing database operations.

**Class:** `AdminComplaintService`

#### Methods

**`getComplaintStats()`** → `Future<Map<String, int>>`
- Fetches all complaints and counts by status: `pending`, `in progress`, `resolved`
- Returns a map: `{'pending': n, 'in_progress': n, 'resolved': n}`

**`getAllComplaints({categoryName?})`** → `Future<List<Complaint>>`
- Fetches all rows from `complaints` table ordered by `created_at` descending
- Maps each row to a `Complaint` object via `Complaint.fromMap()`

**`updateComplaintStatus({complaintId, newStatus, comment?})`**
- Updates `complaints.status` for the given `complaintId`
- If `comment` is provided and non-empty, inserts a new row into `comments` table:
  ```dart
  {'complaint_id': id, 'user_id': adminId, 'content': comment}
  ```

**`getAuditLog()`** → `Future<List<Map<String, dynamic>>>`
- Fetches `title`, `status`, `updated_at`, `user_id` from `complaints`
- Ordered by `updated_at` descending
- Used by `AuditLogScreen` to display complaint history

## Screens

### `login_screen.dart`

**Purpose:** Entry point for all users. Authenticates credentials and routes to the correct dashboard based on role.

**State:** `_LoginScreenState` (StatefulWidget)

**Controllers:** `_emailController`, `_passwordController`

**Key Logic — `_handleLogin()`:**
1. Calls `AuthService.signIn(email, password)`
2. Fetches role via `AuthService.getUserRole()`
3. Routes to `/admin_dashboard` if email is `dev1@gmail.com` OR role is `'admin'`
4. Otherwise routes to `/student_home`
5. Shows a `SnackBar` on success or failure

**UI Highlights:**
- Navy blue AppBar with title `"FAST STUDENT COMPLAINT PORTAL"`
- `Icons.lock_person_rounded` hero icon
- Rounded `TextField` inputs for email and password
- Full-width `ElevatedButton` with navy blue styling
- `TextButton` link to the register screen
- `CircularProgressIndicator` shown during loading state

### `student_home_screen.dart`

**Purpose:** Main landing page for logged-in students. Shows their personal complaint history with real-time status updates.

**State:** `_StudentHomeScreenState` (StatefulWidget)

**Real-time Stream:**
```dart
_supabase
  .from('complaints')
  .stream(primaryKey: ['id'])
  .eq('user_id', currentUser.id)
  .order('created_at', ascending: false)
```
Filtered by `user_id` (Row Level Security), so students only see their own complaints.

**UI Highlights:**
- `StreamBuilder` drives the list — updates automatically without manual refresh
- Each complaint card shows title, status badge (orange = Pending, green = Resolved), and a leading `CircleAvatar` icon
- Tapping a card navigates to `ComplaintDetailScreen`
- `FloatingActionButton` (navy blue `+` icon) opens `NewComplaintScreen`
- Logout `IconButton` in AppBar calls `signOut()` and redirects to `/login`

### `new_complaint_screen.dart`

**Purpose:** Complaint submission form for students.

**State:** `_NewComplaintScreenState` (StatefulWidget)

**Controllers:** `_titleController`, `_descController`

**State Variable:** `_selectedCatId` (`int?`) for category dropdown

**Key Logic — `_submit()`:**
```dart
await Supabase.instance.client.from('complaints').insert({
  'title': _titleController.text,
  'description': _descController.text,
  'category_id': _selectedCatId,
  'user_id': currentUser.id,
  'status': 'pending',
});
```
After insert, pops back to `StudentHomeScreen`.

**Categories Available:**

| ID | Label |
|---|---|
| 1 | Cafeteria |
| 2 | Transport |

### `complaint_detail_screen.dart`

**Purpose:** Read-only view of a single complaint for the student. Fetches and displays any admin feedback from the `comments` table.

**Type:** `StatelessWidget`

**Receives:** `Map<String, dynamic> complaint` as constructor argument

**Admin Feedback Fetch — `_getAdminFeedback()`:**
```dart
Supabase.instance.client
  .from('comments')
  .select()
  .eq('complaint_id', complaint['id'])
  .maybeSingle()
```
Returns `null` if no admin comment exists yet.

**UI Highlights:**
- Title displayed in uppercase with a colored status badge
- "STUDENT DESCRIPTION" section with the complaint body
- `FutureBuilder` for admin feedback section:
  - If no feedback: grey italic placeholder text
  - If feedback exists: blue-tinted card with `Icons.admin_panel_settings` icon and `"OFFICIAL ADMIN FEEDBACK"` label

### `admin_action_screen.dart`

**Purpose:** Decision-making screen for admins. Allows changing complaint status and submitting official feedback.

**State:** `_AdminActionScreenState` (StatefulWidget)

**Receives:** `Complaint complaint` object

**Status Options:**
```dart
['pending', 'in progress', 'resolved']
```

**Key Logic — `_handleUpdate()`:**
1. Validates that at least one change was made (status or comment)
2. Calls `AdminComplaintService.updateComplaintStatus()`
3. On success: shows green `SnackBar` and calls `Navigator.pop(context, true)`
4. On error: shows red `SnackBar`

**UI Highlights:**
- Displays complaint title and description at the top
- `DropdownButtonFormField` for status selection (pre-filled with current status)
- Multi-line `TextField` for admin feedback (4 lines, hint text included)
- Full-width navy blue `ElevatedButton` labeled "Update Complaint"
- Loading indicator while async update runs

## Routing Summary

```
/login
  ├── (admin email or role == 'admin') → /admin_dashboard
  │     ├── tap complaint → AdminActionScreen (full-screen push)
  │     └── AppBar action → /audit_log
  └── (student) → /student_home
        ├── FAB → NewComplaintScreen (push)
        └── tap complaint → /complaint_detail (dynamic route)

/register ← linked from /login
```

## Database Tables Used

| Table | Used By | Purpose |
|---|---|---|
| `profiles` | `AuthService` | Role lookup (admin vs student) |
| `complaints` | All screens & services | Central complaint store |
| `comments` | `ComplaintDetailScreen`, `AdminComplaintService` | Admin feedback per complaint |

> **Row Level Security (RLS):** Students can only read/write their own rows in `complaints`. Admins have global visibility. This is enforced at the Supabase/PostgreSQL level.
